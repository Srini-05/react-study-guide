Prerequisites
Before diving into React, you need a solid foundation in HTML, CSS, and modern JavaScript (ES6+). These are the building blocks that React is built upon.
// Main Question
What JavaScript ES6 features are essential to understand before learning React, and why does React rely on them so heavily?
React relies on modern ES6+ JavaScript features because they make component code cleaner and more expressive. The key features are:
  • Arrow functions — Used in event handlers, callbacks, and functional components. They preserve this context automatically.
  • Destructuring — Props and state are often destructured: const { name, age } = props
  • Spread/Rest operators — Used for immutable state updates: setState({...state, count: 1})
  • Template literals — Dynamic class names and strings
  • Modules (import/export) — Every React file uses ES module syntax
  • Promises & async/await — Required for API calls in useEffect
  • Array methods — map() for rendering lists, filter() for conditional data, reduce() for transformations
// Concept
HTML & CSS Fundamentals
React outputs HTML to the DOM. You need to understand semantic HTML tags, CSS selectors, Flexbox & Grid layout, and box model basics before React makes full sense.
// Concept
JS Essentials
  • Variables: let, const
  • Functions & arrow functions
  • Arrays, Objects, Destructuring
  • Spread ... operator
  • Promises, async/await
  • map, filter, reduce

🟢 Beginner Topic 02 / 20
Introduction to React
React is a JavaScript library for building user interfaces, created by Facebook. It uses a component-based architecture and a virtual DOM to create fast, interactive SPAs.
// Main Question
What is the Virtual DOM and how does React use it to make UI updates more efficient than direct DOM manipulation?
The Virtual DOM is a lightweight JavaScript representation (copy) of the actual DOM tree. When state or props change in a React component:

1. React creates a new Virtual DOM tree reflecting the updated state.
2. It diffs the new tree against the previous Virtual DOM (reconciliation).
3. Only the actual DOM nodes that changed are updated — this is called selective re-rendering.

Direct DOM manipulation is expensive because reading/writing the DOM triggers reflows and repaints. React's approach batches updates and minimizes DOM operations, resulting in significantly better performance in complex, data-heavy UIs.
// Concept
SPA (Single Page Application)
A SPA loads a single HTML file and dynamically updates content via JavaScript — no full page reloads. React handles routing and view rendering entirely client-side, giving a native app-like feel.
// Concept
Component-Based Architecture
React UIs are built from small, reusable pieces called components. Each component manages its own logic and rendering. You compose complex UIs by nesting components, just like HTML elements.

🟡 Core Topic 03 / 20
React Basics
JSX, components, props, and rendering are the fundamental building blocks of every React application. Mastering these is non-negotiable.
// Main Question
What is JSX, and how does Babel transform it into valid JavaScript that browsers can understand?
JSX (JavaScript XML) is a syntax extension that lets you write HTML-like code inside JavaScript. Browsers cannot execute JSX directly — it's compiled by Babel (or Vite/Webpack) into regular React.createElement() calls.

Example: 
// JSX
const el = <h1 className="title">Hello</h1>;

// Compiled output
const el = React.createElement("h1", { className: "title" }, "Hello");
Key rules for JSX:
• Must return a single root element (or use <Fragment>)
• Use className instead of class
• Self-close empty tags: <img />
• JavaScript expressions inside { } curly braces
// Concept
Props (Properties)
Props are read-only data passed from a parent component to a child. They are like function arguments for components. A child cannot modify its own props — data flows one way (top-down).
// Concept
Conditional Rendering
React can conditionally render elements using JavaScript operators: ternary ? :, logical AND &&, or if/else before the return statement. This makes dynamic UIs straightforward.

🟡 Core Topic 04 / 20
State Management Basics
State is the memory of your component. When state changes, React re-renders the component. useState is the primary hook for managing local state.
// Main Question
What is the difference between controlled and uncontrolled components in React forms? When would you use each?
Controlled Components: React state is the single source of truth. The input's value is tied to useState and updated via an onChange handler. React always knows the input's current value.
const [name, setName] = useState("");
<input value={name} onChange={e => setName(e.target.value)} />
Uncontrolled Components: The DOM itself manages the state. You access values using useRef or directly via the DOM. Less code, but React loses control of the value.

Use controlled when: You need instant validation, conditional submission logic, or want to sync the form with other state.
Use uncontrolled when: You have a simple form and don't need live access to values (e.g., file uploads).
// Concept
useState Hook
const [count, setCount] = useState(0) — Returns the current state value and a setter function. Calling the setter triggers a re-render. State updates are asynchronous and batched by React.
// Concept
Event Handling
React uses synthetic events that wrap native browser events. Common handlers: onClick, onChange, onSubmit, onKeyDown. Always pass a function reference, not a function call.

🟡 Core Topic 05 / 20
React Hooks
Hooks let you use state and lifecycle features in functional components. They are arguably the most important concept in modern React development.
// Main Question
Explain the useEffect hook. What does the dependency array control, and what happens when you return a function from useEffect?
useEffect runs side effects after a component renders (API calls, subscriptions, DOM manipulation).

Dependency array behavior:
• useEffect(() => {...}) — Runs after every render
• useEffect(() => {...}, []) — Runs once on mount (like componentDidMount)
• useEffect(() => {...}, [count]) — Runs when count changes

Cleanup function (the returned function): Runs before the next effect or before the component unmounts. Used to cancel subscriptions, clear timers, cancel API requests:
useEffect(() => {
  const timer = setInterval(() => tick(), 1000);
  return () => clearInterval(timer); // cleanup
}, []);
// Concept
useRef
Returns a mutable object with a .current property. Used to directly access DOM nodes (<input ref={inputRef}/>) or to store values that persist across renders without triggering a re-render.
// Concept
useMemo & useCallback
useMemo memoizes an expensive computed value. useCallback memoizes a function reference. Both take a dependency array and only recompute when dependencies change — preventing unnecessary re-renders.

🟠 Intermediate Topic 06 / 20
Styling in React
React offers multiple styling strategies — from plain CSS to utility-first frameworks. Choosing the right approach depends on project size, team conventions, and performance requirements.
// Main Question
What are CSS Modules and how do they solve the problem of global CSS class name collisions in large React apps?
CSS Modules are CSS files where all class names are locally scoped by default. When you import a CSS Module, Vite/Webpack transforms class names into unique hashed identifiers, guaranteeing no collisions.
/* Button.module.css */
.btn { background: blue; color: white; }

// Button.jsx
import styles from './Button.module.css';
<button className={styles.btn}>Click</button>
// Renders as: class="Button_btn__xZq3k"
This means a .btn class in Button.module.css is completely isolated from a .btn class in Header.module.css — solving the global namespace problem of plain CSS without needing complex naming conventions like BEM.
// Concept
Tailwind CSS
Utility-first CSS framework. Instead of writing custom CSS, you compose styles using predefined classes directly in JSX: className="flex items-center gap-4 bg-blue-500 text-white rounded-lg". Extremely fast for development.
// Concept
Styled Components
A CSS-in-JS library where you write actual CSS inside JavaScript template literals. Styles are scoped to components, support dynamic values via props, and generate unique class names automatically.

React Router
React Router enables client-side navigation, letting you build multi-page experiences without full page reloads. It's the standard routing solution for React SPAs.
// Main Question
What are dynamic routes in React Router and how do you access route parameters inside a component?
Dynamic routes use URL segments prefixed with : as placeholders that match any value: 
<Route path="/users/:userId" element={<UserProfile />} />
Inside the component, you access the parameter using the useParams() hook: 
import { useParams } from 'react-router-dom';

function UserProfile() {
  const { userId } = useParams();
  // fetch user data using userId
  return <div>User: {userId}</div>;
}
You can also use useNavigate() for programmatic navigation and useSearchParams() for query string access (?search=react).
// Concept
Link vs NavLink
<Link> replaces <a> tags for client-side navigation. <NavLink> is identical but adds an active class/style automatically when the route matches — perfect for navigation menus.
// Concept
Nested Routes & 404
Child routes render inside a parent's <Outlet /> component. A catch-all path="*" route handles 404 Not Found pages. Nested routes keep layout components persistent across child views.

API & Data Fetching
Most React apps need to communicate with a backend. Understanding how to fetch data, manage loading/error states, and handle async operations is critical.
// Main Question
How do you properly fetch data in a React component using useEffect? How do you handle loading, error, and cancellation?
The standard pattern uses useEffect with three state variables: 
function UserList() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();
    
    async function fetchUsers() {
      try {
        const res = await fetch('/api/users', { 
          signal: controller.signal 
        });
        if (!res.ok) throw new Error('Failed to fetch');
        const data = await res.json();
        setUsers(data);
      } catch (err) {
        if (err.name !== 'AbortError') setError(err.message);
      } finally {
        setLoading(false);
      }
    }

    fetchUsers();
    return () => controller.abort(); // cleanup
  }, []);

  if (loading) return <Spinner />;
  if (error) return <ErrorMessage msg={error} />;
  return <ul>{users.map(u => <li>{u.name}</li>)}</ul>;
}
The AbortController cancels the fetch when the component unmounts, preventing state updates on unmounted components.
// Concept
Fetch vs Axios
The native fetch() API is built into browsers. Axios is a third-party library with automatic JSON parsing, request/response interceptors, better error handling, and request cancellation. Both are valid choices.
// Concept
REST APIs
REST uses standard HTTP methods: GET (read), POST (create), PUT/PATCH (update), DELETE (remove). APIs return JSON. Always handle HTTP error status codes (4xx, 5xx) explicitly.

Forms & Validation
Handling user input efficiently using controlled and uncontrolled components, and validating data with modern libraries like React Hook Form.
// Main Question
What is the advantage of using React Hook Form over traditional controlled components?
Traditional controlled components require a state variable for every input and re-render the entire component on every keystroke. This can cause performance issues in large forms.

React Hook Form uses uncontrolled components internally (via refs) to gather values, preventing unnecessary re-renders. It also provides built-in validation, error handling, and integrates seamlessly with validation schemas like Zod or Yup.
// Concept
Controlled vs Uncontrolled
Controlled components sync input value with React state (useState). Uncontrolled components store value in the DOM and are accessed via useRef.
// Concept
Form Validation
Client-side validation ensures data correctness before submission. Can be manual (checking state on submit) or schema-based (Yup/Zod) for robust type safety.

🔵 Advanced Topic 10 / 20
Advanced State Management
As applications grow, local state becomes insufficient. Context API, Redux, and Redux Toolkit solve the problem of sharing state across many components.
// Main Question
What is prop drilling, and how does the Context API solve it? When would you choose Redux over Context?
Prop drilling is when you pass props through many intermediate components that don't use them — just to get data to a deeply nested child. This creates tight coupling and makes refactoring painful.

Context API solves this by creating a global "broadcast" channel. Any component in the tree can consume context values directly without intermediate passing: 
const ThemeContext = createContext('light');
// Provider wraps the tree
<ThemeContext.Provider value="dark">
  <App />
</ThemeContext.Provider>
// Any descendant can consume it
const theme = useContext(ThemeContext);
Choose Redux when:
• State logic is very complex with many actions
• You need Redux DevTools time-travel debugging
• Multiple teams need consistent state update patterns
• Performance is critical (Context re-renders all consumers)
// Concept
Redux Toolkit (RTK)
The modern, official way to use Redux. RTK eliminates boilerplate with createSlice() (combines actions + reducers), configureStore(), and built-in Immer for immutable updates written like mutations.
// Concept
useReducer
Similar to useState but for complex state logic. Takes a reducer function and initial state. Dispatch actions to trigger state transitions — great for state machines and complex forms before reaching for Redux.

Performance Optimization
React is fast by default, but large apps can slow down. Understanding memoization, code splitting, and lazy loading prevents unnecessary re-renders and reduces bundle sizes.
// Main Question
Explain React.memo, useMemo, and useCallback. How do they each prevent unnecessary re-renders?
React.memo — HOC that wraps a component. Prevents re-render if props haven't changed (shallow comparison). Use for pure components that receive the same props frequently: 
const MemoizedChild = React.memo(({ count }) => <div>{count}</div>);
useMemo — Memoizes an expensive computed value. Only recomputes when dependencies change: 
const expensiveResult = useMemo(() =>
  heavyComputation(data), [data]);
useCallback — Memoizes a function. Critical when passing callbacks to React.memo children, because functions are recreated on every render otherwise: 
const handleClick = useCallback(() => {
  doSomething(id);
}, [id]); // stable reference
Key insight: Don't premature-optimize. Only use these when you've measured a real performance issue with React DevTools Profiler.
// Concept
Code Splitting
Instead of one giant bundle, split code into chunks loaded on demand. React's React.lazy() with Suspense enables component-level code splitting — reducing initial load time significantly.
// Concept
Lazy Loading
const LazyPage = React.lazy(() => import('./HeavyPage')) loads the component only when it's first rendered. Wrap with <Suspense fallback={<Loader/>}> to show a placeholder during loading.

🔵 Advanced Topic 12 / 20
Custom Hooks
Custom Hooks extract reusable stateful logic from components into separate functions, enabling you to share behavior without duplicating code across your application.
// Main Question
What is a custom hook? Write a useFetch custom hook that handles loading, error, and data state for any URL.
A custom hook is simply a JavaScript function whose name starts with "use" and that may call other hooks internally. It lets you extract component logic into a reusable function: 
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();
    setLoading(true);

    fetch(url, { signal: controller.signal })
      .then(res => {
        if (!res.ok) throw new Error(res.statusText);
        return res.json();
      })
      .then(data => { setData(data); setLoading(false); })
      .catch(err => {
        if (err.name !== 'AbortError') {
          setError(err.message); setLoading(false);
        }
      });

    return () => controller.abort();
  }, [url]);

  return { data, loading, error };
}

// Usage in any component:
const { data, loading, error } = useFetch('/api/posts');
The custom hook is now reusable across any component that needs to fetch data.

Advanced React Concepts
HOCs, render props, portals, error boundaries — these patterns solve architectural challenges in large React applications and appear frequently in advanced interviews.
// Main Question
What is a Higher Order Component (HOC)? How does it differ from custom hooks for code reuse?
A Higher Order Component is a function that takes a component and returns a new enhanced component. It's a pattern for code reuse at the component level: 
// HOC: adds loading behavior to any component
function withLoading(WrappedComponent) {
  return function({ isLoading, ...props }) {
    if (isLoading) return <Spinner />;
    return <WrappedComponent {...props} />;
  };
}
const UserListWithLoading = withLoading(UserList);
HOC vs Custom Hooks:
• HOCs wrap at the component tree level — they affect rendering
• Custom hooks share stateful logic — they don't control what renders
• Custom hooks are generally preferred in modern React (less nesting, easier to understand)
• HOCs are still used in libraries like React-Redux's connect() and React Router's older API

Error Boundaries are class components that catch JS errors in their child tree, preventing the whole app from crashing. They use componentDidCatch and getDerivedStateFromError.
// Concept
Portals
Render a component outside its DOM parent — useful for modals, tooltips, and dropdowns that need to escape overflow:hidden or stacking context issues. Use ReactDOM.createPortal(child, domNode).
// Concept
Reconciliation
React's algorithm for comparing old vs new Virtual DOM trees (diffing). It uses the element type and key prop as heuristics to minimize DOM operations. Understanding it explains why key matters in lists.

🔴 Expert Topic 14 / 20
React Authentication
Authentication is about verifying who a user is. In React apps, this typically involves JWT tokens, protected routes, and managing auth state globally.
// Main Question
How does JWT (JSON Web Token) authentication work in a React application, from login to accessing protected routes?
JWT Authentication Flow:

1. Login: User submits credentials. Backend validates and returns a JWT token (3-part: header.payload.signature)
2. Storage: Store the token in localStorage or HTTP-only cookies (more secure)
3. Requests: Attach token to API requests via Authorization header:
headers: { 'Authorization': `Bearer ${token}` }
4. Auth context: Use React Context to share auth state globally: 
const AuthContext = createContext();
// Provide: { user, token, login, logout }
5. Protected Routes: Check authentication before rendering: 
function ProtectedRoute({ children }) {
  const { user } = useContext(AuthContext);
  return user ? children : <Navigate to="/login" />;
}
6. Logout: Clear token from storage and reset auth context state.

Security note: HTTP-only cookies prevent XSS attacks from stealing tokens (localStorage is vulnerable to XSS).

Testing in React
Testing ensures your components work correctly and regressions are caught early. The React Testing Library philosophy: test behavior, not implementation details.
// Main Question
What is the difference between unit testing, integration testing, and end-to-end (E2E) testing in a React context?
Unit Testing — Tests a single component or function in isolation. Mocks all dependencies.
// Testing that a Button renders its label
test('renders label', () => {
  render(<Button label="Click me" />);
  expect(screen.getByText('Click me')).toBeInTheDocument();
});
Integration Testing — Tests how multiple components work together. May test a form submission flow, routing, or context interactions. This is where React Testing Library excels — it renders actual DOM:
test('submits form and shows success', async () => {
  render(<LoginForm />);
  userEvent.type(screen.getByLabelText('Username'), 'john');
  userEvent.click(screen.getByRole('button', { name: /login/i }));
  await waitFor(() => 
    expect(screen.getByText('Welcome!')).toBeInTheDocument()
  );
});
E2E Testing — Tests the full application in a real browser (Cypress, Playwright). Slowest but most realistic. Tests the entire stack: frontend + API + database.

The Testing Trophy: Most value comes from integration tests. Unit tests for complex logic, E2E for critical user flows.

Project Architecture

// Main Question
What is "Separation of Concerns" in the context of a React application?
Separation of Concerns (SoC) means dividing an application into distinct sections, each addressing a specific concern.

In React, this typically involves separating UI logic (presentation) from business logic (data fetching, state manipulation). Common patterns include:
• Custom hooks to encapsulate state and side effects.

• Feature-based folder structures where all related files (components, hooks, styles, tests) are grouped together.
// Concept

Organizing code by feature (e.g., src/features/auth) rather than type (e.g., src/components, src/hooks) scales better for large apps.
// Concept

Building components that are small, focused, and accept configuration via props (like Compound Components or Render Props) maximizes reusability.

Backend Integration
React apps rarely work in isolation. Integrating with REST APIs, GraphQL, and handling file uploads are essential full-stack skills.
// Main Question
What is GraphQL and how does it differ from REST APIs? What are its advantages when used with React?
GraphQL is a query language for APIs where the client specifies exactly what data it needs in a single request. Unlike REST:

REST: Multiple endpoints, each returning a fixed shape
GET /users/1 → returns ALL user fields (may include unneeded data)
GET /users/1/posts → separate request for related data

GraphQL: One endpoint, client defines the shape
query {
  user(id: "1") {
    name
    email
    posts { title }
  }
}
Advantages with React:
• No over-fetching — Get only what you render
• No under-fetching — Related data in one round-trip
• Type safety with TypeScript code generation
• Libraries like Apollo Client or TanStack Query integrate seamlessly with React state

🟣 Full-Stack Topic 18 / 20
Next.js Full Stack
Next.js is a React framework that adds SSR, SSG, file-based routing, and API routes. It solves React's SEO limitations and enables full-stack development in a single codebase.
// Main Question
What is the difference between Server-Side Rendering (SSR), Static Site Generation (SSG), and Client-Side Rendering (CSR)? When would you use each?
CSR (Client-Side Rendering) — Default React behavior. Browser downloads a blank HTML shell, then JavaScript builds the UI. SEO is poor (crawlers may miss JS-rendered content). Fast after initial load.

SSR (Server-Side Rendering) — On each request, the server renders the HTML and sends it to the browser. In Next.js, use getServerSideProps(). Best for pages with frequently changing data (user dashboard, personalized content). Good SEO.

SSG (Static Site Generation) — HTML is generated at build time and served as static files. Fastest possible delivery via CDN. Use getStaticProps() in Next.js. Best for content that doesn't change often: blogs, docs, marketing pages.

ISR (Incremental Static Regeneration) — Next.js hybrid. Pages are statically generated but can be revalidated in the background after a set interval. Best of both worlds for most cases. 
// SSR: runs on every request
export async function getServerSideProps(context) {
  const data = await fetchUserData(context.req.cookies.token);
  return { props: { data } };
}

// SSG: runs once at build time
export async function getStaticProps() {
  const posts = await fetchBlogPosts();
  return { props: { posts }, revalidate: 60 }; // ISR
}

// Main Question
Walk through the process of deploying a React (Vite) application to Vercel. What does the build process actually produce?
Build process: Running npm run build with Vite:
1. Transpilation: JSX and modern JS compiled to browser-compatible JavaScript
2. Bundling: All modules combined into chunks
3. Minification: Whitespace/comments removed, variable names shortened
4. Tree-shaking: Unused code eliminated
5. Output: A dist/ folder with static files: index.html, JS chunks, CSS, assets

Deploying to Vercel:
1. Push your code to GitHub
2. Connect repo to Vercel (vercel.com)
3. Vercel auto-detects Vite, runs the build command
4. Sets up global CDN, HTTPS, and automatic preview deployments on every PR
5. Set environment variables in Vercel dashboard (never commit .env)

For Next.js: Vercel is the native platform — SSR, API routes, and ISR all work automatically without extra configuration.

Netlify vs Vercel vs Firebase
Vercel: Best for React/Next.js, excellent DX, SSR support.
Netlify: Great for static sites, forms, serverless functions.
Firebase: Google's platform — great when using Firebase backend (Firestore, Auth).

// Main Question
What React concepts does an E-commerce application exercise, and what are the most challenging technical decisions you'd face building one?
An e-commerce app exercises almost every React concept:

State Management: Cart state needs to be global (Context or Redux). Product data comes from an API. User auth state needs to persist across sessions.

Routing: Product listing, detail pages (/products/:id), checkout flow, protected order history route.

Performance: Product images need lazy loading. Large product lists need virtualization. Search needs debouncing.

Challenging decisions:
1. Cart persistence — localStorage sync across tabs, merge guest cart on login
2. Optimistic UI — Show cart update immediately before server confirms
3. Payment integration — Stripe Elements, handling success/failure redirects
4. Search & filtering — URL params vs state, debounced API calls
5. Image optimization — Multiple sizes, WebP format, lazy loading
6. SEO — Product pages need SSR/SSG (consider Next.js)

Building this forces you to make real architectural decisions under real constraints — the best learning experience in this roadmap.
